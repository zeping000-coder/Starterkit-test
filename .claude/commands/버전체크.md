## 입력 형식

`$ARGUMENTS`는 확인하고 싶은 라이브러리 이름 하나입니다.
예: `/버전체크 lightweight-charts`, `/버전체크 prisma`, `/버전체크 next`

값이 없다면, 지금 작업 중인 코드에서 어떤 라이브러리를 다루고 있는지 먼저 물어보세요.

## 왜 필요한가

이 프로젝트(`web/`)는 최신 메이저 버전으로 고정된 라이브러리가 많아서, 흔히 알려진 옛날 API를 그대로 가정하면 틀린 코드를 만들게 됩니다. 예를 들어 `chart.addLineSeries()`는 lightweight-charts v5에서 제거되고 `chart.addSeries(LineSeries, options)`로 바뀌었고, Prisma 7은 `datasource url`이 `schema.prisma`가 아니라 `prisma.config.ts`에 있습니다. **"비유하자면, 몇 년 전에 배운 지도를 들고 새로 지어진 건물에 찾아가는 것과 같아요 — 겉모습은 비슷해 보여도 입구 위치가 바뀌었을 수 있어요."**

## 작업 순서

1. **설치된 버전 확인**
   - `web/package.json`의 `dependencies` / `devDependencies`에서 `$ARGUMENTS`와 일치하는 패키지의 정확한 버전을 확인
   - 못 찾으면 사용자에게 정확한 패키지명을 되물어보기

2. **로컬 문서 우선 확인 (Next.js인 경우 필수)**
   - `web/AGENTS.md`에 명시된 대로, Next.js 관련 작업이면 반드시 `web/node_modules/next/dist/docs/`를 먼저 읽기
   - 다른 라이브러리도 `web/node_modules/<패키지>/README.md`나 `CHANGELOG.md`가 있으면 함께 확인

3. **Context7 MCP로 최신 문서 조회**
   - `resolve-library-id`로 정확한 라이브러리 ID를 찾은 뒤, 1번에서 확인한 설치 버전에 맞춰 `query-docs`로 실제 사용하려는 API/문법을 질의
   - 결과를 바탕으로 "이 버전에서는 이렇게 써야 한다"를 요약해서 사용자에게 먼저 보여주기

4. **breaking change 여부 요약**
   - 이전에 흔히 쓰이던 옛날 API와 달라진 부분이 있다면 짧게 표로 정리 (예: "예전엔 X였는데, 이 버전에서는 Y")
   - CLAUDE.md에 이미 언급된 항목(Next.js 16 Cache Components 비활성화, Prisma 7 어댑터 방식, base-nova가 Base UI 기반 등)과 겹치면 그 내용도 함께 상기시키기

5. **확인 후에만 코드 작성 진행**
   - 위 확인 없이 바로 코드를 쓰지 말 것. 문서 확인 결과를 먼저 사용자에게 보여주고, 그 다음에 실제 구현 작업으로 넘어가기
